# Agent Spec

## Overview
Deadlock Coach should become an ADK 2.0 multi-agent product centered on a root LLM coach agent.

The root `coach_agent` is the primary conversational interface. It should interpret the user's question, decide which specialist analyst nodes or tools to call, gather evidence from local and external data sources, consult a local knowledge base when relevant, and return a grounded coaching answer.

The first version should be a prototype-first architecture with:

- a root `coach_agent`
- specialist analyst agents or workflow nodes for:
  - player profile analysis
  - hero pool analysis
  - build and item timing analysis
  - player-vs-pro or player-vs-meta comparison
  - report generation
- a local knowledge-base retrieval tool that reads project knowledge files and returns relevant excerpts for the active question
- local normalized SQLite-backed data access as the main fast path for player-specific evidence

The system should support both:

- conversational coaching in the `Coach` tab
- structured routes that later power product tabs such as Hero Lab, Builds, Pro Mirror, and Reports

The ADK design should favor explicit workflows and specialist nodes over a single monolithic prompt.

## Example Use Cases

### 1. Hero overview
Input:
`What heroes do I usually play?`

Expected behavior:
- root coach agent routes to hero-pool analysis
- hero analyst reads recent player sample
- response summarizes recent hero mix with counts and identifies the main hero

### 2. Build advice
Input:
`What do I usually build on Billy?`

Expected behavior:
- root coach agent routes to build analysis
- build analyst reads recent item path and timing data
- response summarizes the most repeated build spine and key item timings

### 3. Coaching prioritization
Input:
`What should I focus on right now?`

Expected behavior:
- root coach agent gathers player form, hero sample concentration, item timing anchors, and optional knowledge-base heuristics
- response gives a concise recommendation with evidence and a clear next step

### 4. Comparison
Input:
`Compare my Billy build to better players`

Expected behavior:
- root coach agent routes to comparison analysis
- comparison analyst combines player-specific data with meta or cohort data
- response clearly separates personal sample from external comparison

### 5. Knowledge-grounded question
Input:
`What should I remember when I play Billy into Shiv?`

Expected behavior:
- root coach agent queries the knowledge-base tool
- if matching internal notes exist, it uses them alongside player data
- response indicates when a point comes from knowledge files rather than telemetry

## Tools Required

### 1. Local player data tool
Purpose:
- Read normalized player-specific match, hero, and item data from local SQLite or local service layers.

Details:
- Primary source for player-specific evidence
- Should expose recent matches, hero performance, item timings, and summary stats

Auth:
- None for local reads

### 2. Deadlock API integration tool
Purpose:
- Fetch or refresh player-specific match and account data from deadlock-api when local data is stale or missing

Details:
- Used for match history, player-specific metadata, and sync flows

Auth:
- API key or project-configured auth if required later

### 3. Statlocker integration tool
Purpose:
- Fetch broader meta, build, item, and cohort information that is better suited for comparison and trend analysis

Details:
- Used for best-build context, item popularity, hero usage trends, and meta comparisons

Auth:
- API key if available later

### 4. Knowledge-base retrieval tool
Purpose:
- Search local knowledge files and return relevant passages or notes

Details:
- Initial implementation should support simple local file retrieval from a dedicated knowledge folder
- Later versions can add chunking, tagging, and ranking

Auth:
- None for local files

### 5. Report assembly tool
Purpose:
- Convert collected evidence into structured outputs such as reports, summaries, and experiment plans

Details:
- Used by report and workflow-oriented nodes

Auth:
- None

## Constraints & Safety Rules

- Never invent player stats, item timings, hero usage, or win rates
- Always distinguish between:
  - player-specific telemetry
  - external meta or cohort data
  - local knowledge-base guidance
- Prefer “I do not know yet” over fabricated certainty
- Do not present a recommendation as evidence-backed unless it is grounded in actual retrieved data
- If a question needs external comparison data that is unavailable, say so clearly
- Keep prototype responses concise, direct, and useful
- The `Coach` tab remains the main interaction surface; deeper analyst routes should support it rather than replace it

## Success Criteria

- The root coach agent can answer common player questions using real LLM reasoning plus tool calls
- The system can route hero, build, comparison, and report questions to the right specialist path
- The knowledge-base tool can retrieve local notes for relevant questions
- Responses clearly separate evidence source types
- The architecture is ready to later power multiple product tabs without rewriting the core coaching agent

## Reference Samples

- `deep-search`
  - useful for multi-step grounded synthesis and evidence-backed reporting
- `memory-bank`
  - useful later if the coach should remember user preferences or long-term coaching context
- `rag-agent-search` or `rag-vector-search`
  - useful later if the knowledge base evolves beyond simple local file retrieval

## Architecture & Sub-Agents

### Root agent
- `coach_agent`
  - primary LLM agent for the Coach tab
  - decides which specialist node, tool, or workflow branch to use

### Specialist nodes or agents
- `player_profile_analyst`
- `hero_pool_analyst`
- `build_analyst`
- `comparison_analyst`
- `report_agent`

### Workflow ideas
Initial workflow routes should include:

- hero overview
- hero recommendation
- build overview
- timing diagnosis
- comparison to stronger players or meta
- focus and next-step coaching
- report and experiment planning

## Data Sources & Auth

- Local SQLite or normalized local storage
  - no auth
- deadlock-api
  - used for player-specific match and account data
- statlocker
  - used for broader meta, cohort, and build intelligence
- local knowledge files
  - stored inside the project and accessed by retrieval tools

## Non-Functional Requirements

- Prototype first, not deployment-first
- Keep the architecture modular so the current rule-based backend can be replaced incrementally
- Make it easy to test specialist routes independently
- Preserve a fast local loop for development and evaluation
